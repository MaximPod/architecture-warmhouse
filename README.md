# Задание 1. Анализ и планирование

### 1. Описание функциональности монолитного приложения
Анализ исходного кода приложения smarthome показал следующее:

**Управление отоплением:**
- функциональное требование "управление отоплением" в приложении не реализовано

**Мониторинг температуры:**
-	функциональное требование "мониторинг температуры" реализовано со следующими ограничениями:
 	+ активный периодический мониторинг датчиков не реализован
	+ опрос датчиков выполняется через обработчики синхронных запросов к api приложения
	+ приложение может связываться только с одним сервисом\контроллером датчиков температуры, указанным в конфигурации приложения
	+ приложение сохраняет значение температуры в БД. История значений не температуры не ведется. 

**Пользователи**  
 Пользователи как сущность в приложении не реализованы.  
 Регистрация\Идентификация\Аутентификация\Авторизация пользователей не реализована

**Работа с датчиками**  
В приложении реализована только работа с датчиками температуры

**Коллекция датчиков**  
В приложении реализован CRUD для хранения коллекции датчиков в СУБД Postgresql  
Реализованы такие функции как: 
- создание, 
- получение списка всех датчиков с обязательным обновлением данных для датчиков температуры от сервиса температуры,  
- получение параметров одного датчика по ID с обязательным обновлением данных для датчика температуры от сервиса температуры
- обновление параметров датчика по его ID в БД
- обновление температуры и статуса датчика в БД по его ID
- удаление датчика по его ID из БД
 
**UI интерфейс для пользователей**  
- UI интерфейс в приложении не реализован

### 2. Анализ архитектуры монолитного приложения
#### Общие данные
+ Имя приложения: smarthome  
+ Язык разработки: Golang v 1.22  
+ Зависимости:
	- Логирование: стандартное пакет log
	- Метрики: отсутствуют
    - Хранение данных: СУБД Postgresql
	- Конфигурация: из переменных окружения

#### Архитектурный обзор
**Анализ структуры папок и файлов**  
- Структура папок соответствует одной из общепринятых практик.
- Хранение Vendor не используется.
- Миграции оформлены в виде файла init.sql в корне репозитория
- Код билдится.

**Анализ внешних зависимостей**  
Внешние зависимости представлены библиотеками общей практики, экзотики не обнаружено.

**Анализ внутренных зависимостей и декомпозиции на пакеты**
- Внутренние пакеты не собраны в папку internal
- Главная сущность представлена в виде модели. Модель не имеет поведения.
- Бизнес логика утекла в модуль обрабочиков эндпойнтов api в пакет handlers
- Пакет handlers является адаптером первого уровня. При этом он имеет зависмости от адаптеров второго уровня (db и services). Эти зависимости в текущей реализации сильно связаны.

**Детальная проверка кода**  
**Читаемость и стиль написания**  
- Код читается. Форматирование частично соблюдено.
- Функции и структуры покрыты коментариями
- Комментарии полей структур отсутствуют, в совокупности с отсутствием документации это вызывает дополнительную конгнитивную нагрузку. 

**Обработка ошибок**
- Ошибки либо логируются, либо отдаются клиенту.
- Текст ошибок не содержит полезной детализации для анализа причин ошибки.

**Безопасность**  
- Нет авторизации к эндпойнтам api
- Нет валидации входных параметров api

**Производительность**  
- Бизнес логика реализована не эффективно. 
- Наибольшее изумление вызывает реализация обработчика эндпойнта /api/v1/sensors


**Тестирование**  
- Код не покрыт unit-тестами. 
- Тест кейсы находятся в корне проекта в виде файла коллекции postman  
Причина такого состояния в высокой связанности модулей и нарушении принципов ЧА

**Документирование**  
- Проектное решение (solution) - не представлено
- Документация не реализована
- REST API не собран в спецификацию openapi (swagger)

# Задание 4. Создание и документирование API

### 1. Тип API

В системе используются несколько типов взаимодействия:
- Async Api для междоменного взаимодействия.
- Sync Api для приема данных
- Async Api через REST для фронтендов. 

Причины:
 - Высокий уровень неопределенности в постановке задачи на проектирование
 - Отсутствие экспертизы в предметной области
 - Многовариантность дальнейшего развития системы
 
Асинхронный обмен через шины данных позволяет поддерживать слабую связанность, что обеспечивает возможность развития системы в широком диапазоне направлений

Фронтенды взаимодействуют с сервисом BFF через REST API. Обмен асинхронный по Pub/Sub модели. Отправка данных осуществляется через POST  запрос. Прием данных по SSE потоку (Get event-stream)
